# Phase 0.7: Feature Ordering

## Responsible agents
`product-owner` + `architect` (collaborative)

## Objective
Order features into a coherent implementation plan (epics) that respects both business priorities and technical dependencies.

## Prerequisites
- Phase 0 (Scoping) validated — features and priorities defined
- Phase 0.5 (Design) validated — UI components and layouts specified

## Instructions

You are in **Phase 0.7 — Feature Ordering**. The Product Owner and Architect work together to define the optimal implementation order.

### Step 1: PO inputs — Business priorities
1. Review the MoSCoW prioritization from the spec
2. Identify which features deliver the most user value earliest
3. Define what "testable value" means for each feature group
4. Identify features that are cross-cutting (e.g., design system, expert positioning) and should be distributed across epics rather than concentrated in one

### Step 2: Architect inputs — Technical dependencies
1. Identify foundation features that must come first (e.g., cache layer, auth, data model)
2. Map technical dependencies between features (A requires B to be built first)
3. Identify features that can be parallelized
4. Flag high-risk features that need early investigation (spikes)

### Step 3: Collaborative ordering
1. Group features into epics (max 8-10 epics)
2. Order epics respecting BOTH business value AND technical dependencies
3. Each epic must deliver:
   - Testable user value (not purely technical)
   - A clear "definition of done"
4. Cross-cutting features are distributed across epics, not concentrated in one
5. Identify parallelization opportunities

### Step 4: Document the plan
For each epic, document:
```markdown
## Epic N: [Name]
**Features covered**: F1, F2...
**Prerequisites**: Epic X, Y
**Value delivered**: [what the user can do after this epic]
**Backend**: [API/DB work required]
**Frontend**: [UI work required]
**Tests**: [key E2E tests]
**Relative estimate**: S / M / L / XL
```

### Step 5: Validate with the user
1. Present the epic order with a timeline visualization
2. Explain trade-offs (why X comes before Y)
3. Get explicit validation before proceeding to Phase 1

## Expected deliverable
- Implementation plan document with ordered epics (`specs/[project-name]-implementation-plan.md`)
- Timeline visualization (ASCII or mermaid)
- Dependency graph between epics

## Validation criteria
- [ ] All must-have features are covered in the first epics
- [ ] Technical dependencies are respected (no epic requires a later epic)
- [ ] Each epic delivers testable user value
- [ ] Cross-cutting concerns are distributed, not concentrated
- [ ] High-risk items have early spikes planned
- [ ] User has validated the order
